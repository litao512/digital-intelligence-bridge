# 授权资源快照缓存设计

**目标**

- 为 DIB 宿主增加“当前插件已授权资源快照”运行时层，将资源中心后台授权结果稳定地下发给插件。
- 让插件通过宿主上下文按 `usageKey` 读取资源，避免直接查询后台或直接读取磁盘缓存。
- 为后续审批通过即时生效、撤销授权停止下发、短时离线恢复和资源注入试点提供稳定基础。

**设计范围**

- 本轮聚焦宿主内存快照、磁盘持久化目标形态、插件上下文暴露方式和第一阶段落地顺序。
- 本轮不直接实现宿主代理访问模式；第一阶段默认由宿主向插件下发完整连接参数。
- 本轮不扩展后台授权协议，只消费现有 `GetAuthorizedResourcesAsync()` 能拿到的授权资源结果。

## 一、设计结论

- 授权资源缓存应采用“双层结构”：磁盘持久化快照 + 内存热缓存。
- 第一阶段实现顺序应采用“先内存、后磁盘”，避免同时引入文件持久化和插件注入两类排障点。
- 宿主是资源下发的唯一入口；插件不直接调用后台，不直接读取缓存文件。
- 插件应通过宿主上下文按 `usageKey` 读取资源，优先获取宿主下发的运行时资源；取不到时，试点插件可以回退到本地配置。
- 第一阶段向插件下发“完整连接参数”，并保留未来升级为“资源句柄 + 宿主代理”的演进空间。

## 二、为什么需要授权资源快照缓存

### 1. 为插件提供稳定运行时视图

- 后台授权状态变化频率低，但插件业务执行频率高。
- 宿主需要持有一份当前授权快照，避免插件在高频业务过程中每次都远程查询授权。
- 快照使插件在某个时刻看到的是一致的资源集合，而不是“边执行边变”。

### 2. 维持宿主与插件的职责边界

- 后台是真相源，宿主负责同步和下发，插件负责消费资源并执行业务。
- 没有快照层，插件会倾向于自己实现认证、查询、缓存和异常处理，架构很快散掉。

### 3. 支撑审批、撤销和短时离线

- 审批通过后，宿主同步并更新快照，插件后续即可读取新资源。
- 授权撤销后，宿主更新快照并停止下发资源，避免插件继续使用旧授权。
- 短时后台不可达时，宿主仍可基于最近一次有效快照工作，而不是立即瘫痪。

## 三、备选方案与推荐

### 方案 A：仅内存缓存

- 优点：实现简单，调试路径短。
- 缺点：客户端重启即丢失；无法支持启动即恢复；离线能力弱。

### 方案 B：仅磁盘缓存

- 优点：重启后可恢复。
- 缺点：插件高频读取不顺手；刷新、替换和变更通知都更笨重。

### 方案 C：磁盘持久化 + 内存热缓存

- 优点：启动恢复、运行时高频读取、后续变更通知都更自然。
- 缺点：需要设计双层同步和版本控制。

**推荐结论：**

- 最终形态采用方案 C。
- 第一阶段实现顺序采用“先内存、后磁盘”。

## 四、缓存结构设计

### 1. 站点级快照 `AuthorizedResourceCacheSnapshot`

- 表示当前站点在某一时刻的全部授权资源。
- 建议字段：
  - `SiteId`
  - `SnapshotVersion`
  - `SyncedAt`
  - `ExpiresAt`
  - `Resources`

### 2. 插件级资源集 `AuthorizedPluginResourceSet`

- 表示某个插件当前可用的授权资源集合。
- 建议字段：
  - `PluginCode`
  - `Resources`

### 3. 插件运行时资源 `AuthorizedRuntimeResource`

- 这是宿主真正下发给插件消费的对象。
- 建议字段：
  - `ResourceId`
  - `ResourceCode`
  - `ResourceName`
  - `ResourceType`
  - `UsageKey`
  - `BindingScope`
  - `Version`
  - `Capabilities`
  - `Configuration`

### 4. 结构原则

- 磁盘存“站点级快照”，宿主内存里同时维护站点级快照和插件级索引。
- 插件只看到分配给自己的资源，不看到整站点的全部资源。
- `usageKey` 必须保留，因为它是插件声明资源需求和运行时读取资源的主键。
- 快照必须带版本和时间信息，用于判断新旧和过期状态。

## 五、宿主向插件暴露资源的方式

### 1. 推荐方式

- 宿主在插件上下文 `IPluginHostContext` 中提供资源访问能力。
- 插件通过宿主上下文按 `usageKey` 读取资源。
- 插件不直接访问磁盘快照文件，不直接调用后台接口。

### 2. 推荐接口形态

第一阶段建议在插件宿主上下文增加：

- `GetAuthorizedResources()`
- `TryGetResource(string usageKey, out AuthorizedRuntimeResource? resource)`

其中主路径是 `TryGetResource(usageKey)`，这样插件最不容易滥用宿主下发资源。

### 3. 数据流

1. 宿主同步后台授权资源。
2. 宿主将后台响应转换成站点级授权快照，并写入内存缓存。
3. 插件初始化后，通过宿主上下文读取属于自己的资源集合。
4. 插件在真正执行任务时，按 `usageKey` 获取运行时资源。
5. 审批通过或撤销后，宿主刷新快照，插件后续读取到最新值。

## 六、向插件下发什么内容

### 方案 A：完整连接参数

- 优点：落地快，对现有插件改造小。
- 缺点：插件会接触到运行时凭据，需要控制缓存和注入边界。

### 方案 B：资源句柄 + 宿主代理访问

- 优点：安全边界更强，插件拿不到真实凭据。
- 缺点：实现复杂度高，需要为各类资源提供宿主访问抽象。

**推荐结论：**

- 第一阶段采用“宿主下发完整连接参数”。
- 未来对 OCR、人脸识别、平台级 HTTP 服务等高敏资源，再逐步升级到“资源句柄 + 宿主代理”。

## 七、缓存落地位置与持久化目标

### 1. 目标位置

- 推荐路径：`LocalAppData\\DibClient\\resource-cache\\authorized-resources.json`

### 2. 第一阶段策略

- 先实现内存缓存与插件读取链路。
- 先不落磁盘，避免同时处理持久化、版本兼容和加密存储。
- 待内存链路稳定后，再补磁盘持久化。

### 3. 持久化原则

- 真相源仍然是后台授权接口。
- 本地缓存必须可刷新、可替换、可识别过期。
- 撤销授权后不允许无限期依赖旧缓存。
- 敏感凭据不得长期明文散落在插件目录。

## 八、第一阶段实施边界

### 1. 先做

- 新增 `IAuthorizedResourceCacheService`
- 新增内存态授权快照模型和插件级索引
- 在宿主同步授权资源后写入缓存
- 在插件宿主上下文中增加资源读取接口
- 选择一个试点插件优先读取宿主资源，再回退本地配置

### 2. 暂不做

- 磁盘持久化实现
- 宿主代理资源访问
- 全量插件改造
- UI 侧更多授权状态细节

## 九、推荐试点插件

### 1. `PatientRegistration`

- 已声明 `registration-db`
- 业务路径短，验证“按 `usageKey` 读取数据库资源”比较直接

### 2. `MedicalDrugImport`

- 已声明 `business-db` 和 `sync-target`
- 后续适合验证一个插件消费多类资源

**推荐顺序：**

- 先试点 `PatientRegistration`
- 再扩到 `MedicalDrugImport`

## 十、验收标准

1. 宿主同步授权资源后，内存缓存中能拿到对应插件的资源集。
2. 插件能按 `usageKey` 读取资源。
3. 没有授权时，插件取不到资源而不是误取其他资源。
4. 试点插件在有宿主资源时优先使用宿主资源，在无宿主资源时回退本地配置。
5. 授权刷新后，插件后续读取到最新资源视图。
