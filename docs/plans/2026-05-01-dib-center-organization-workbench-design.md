# DIB 中心单位详情工作台设计

## 背景

DIB 中心已经从发布中心演进为统一控制平面，当前已具备单位、站点、资源、单位授权和站点资源绑定等基础能力。

现有界面按功能模块拆分为多个页签，适合快速落地数据维护，但运营人员在完成一个单位开通时，需要在多个页签之间切换：

1. 新建或编辑单位
2. 将站点绑定到单位
3. 新建或选择资源
4. 给单位授权插件
5. 给单位授权资源
6. 给站点的插件 `usage_key` 绑定具体资源
7. 验证客户端是否能获取资源

随着 DIB 面向医疗、疾控、医保、民政、大数据局等多行业扩展，界面应从“表格维护工具”升级为“按单位治理的运营工作台”。

## 目标

第一阶段目标是新增“单位详情工作台”，让单位成为资源中心和插件授权的主要操作入口。

工作台需要回答四个问题：

- 这个单位有哪些站点？
- 这个单位被授权了哪些插件和资源？
- 这些授权是否已经落到站点资源绑定？
- 当前还有哪些问题会导致客户端拿不到资源？

## 非目标

本阶段不做以下能力：

- 不引入正式标签字典，继续沿用轻量业务标签。
- 不实现完整资源申请审批流。
- 不实现资源密钥维护界面。
- 不重构现有发布、插件、客户端、资产页。
- 不引入复杂策略引擎或跨单位授权协议。

## 产品原则

### 1. 以单位为治理入口

站点绑定单位，单位决定插件和资源授权。因此运营人员进入 DIB 中心后，应能围绕一个单位完成主要治理动作，而不是在“单位、站点、资源、授权、绑定”之间来回找关系。

### 2. 状态优先于字段

界面不能只展示 `Active`、`Inactive`、`Draft` 等原始字段，还要翻译成运营人员能理解的问题：

- 站点未绑定单位
- 单位未授权插件
- 单位未授权资源
- 站点已绑定单位但缺少资源绑定
- 资源已停用但仍被绑定
- 插件已授权但缺少对应 `usage_key` 资源

### 3. 优先补齐闭环

本阶段不追求复杂视觉效果，优先让“一个单位从零开通到客户端可用”的路径变短、错误更少。

## 信息架构

### 单位管理页

保留现有单位列表和新增/编辑能力，在每个单位行增加“进入工作台”动作。

单位列表建议补充以下摘要列：

- 站点数
- 插件授权数
- 资源授权数
- 绑定数
- 问题数

### 单位详情工作台

工作台顶部显示单位基础信息：

- 单位名称
- 单位编码
- 单位类型
- 状态
- 业务标签

顶部摘要指标：

- 站点数
- 已授权插件数
- 已授权资源数
- 已完成资源绑定数
- 待处理问题数

主体分为四个区域。

#### 1. 站点区域

展示该单位下的站点：

- 站点名称
- 站点标识
- 分组
- 客户端版本
- 最近心跳
- 资源绑定状态

需要突出以下异常：

- 未在线或长期未心跳
- 未分组
- 没有任何资源绑定
- 已安装插件但缺少对应资源绑定

提供快捷动作：

- 跳转到站点管理
- 给站点选择分组
- 进入该站点的资源绑定区域

#### 2. 插件授权区域

展示单位已授权插件：

- 插件名称
- 插件编码
- 授权状态
- 到期时间
- 是否已有站点使用

提供动作：

- 新增插件授权
- 停用插件授权
- 查看哪些站点受影响

#### 3. 资源授权区域

展示单位可访问资源：

- 资源名称
- 资源编码
- 资源类型
- 所属单位
- 状态
- 标签

提供动作：

- 新增资源授权
- 停用资源授权
- 查看资源被哪些站点绑定

#### 4. 站点资源绑定区域

选择单位下的某个站点后，显示该站点当前绑定：

- 插件
- `usage_key`
- 资源
- 绑定状态

新增绑定时，候选插件只来自当前单位有效插件授权，候选资源只来自当前单位有效资源授权。

如果不能绑定，界面应直接展示原因：

- 当前站点未绑定单位
- 当前单位没有插件授权
- 当前单位没有资源授权
- 选择的插件未被单位授权
- 选择的资源未被单位授权
- `usage_key` 为空

## 数据流

工作台不新增核心数据模型，复用现有数据：

- `organizations`
- `sites`
- `resources`
- `organization_plugin_permissions`
- `organization_resource_permissions`
- `resource_bindings`
- `plugin_packages`

前端加载流程：

1. 加载全部单位、站点、资源、插件包、单位授权、资源绑定。
2. 根据选中单位过滤站点、插件授权和资源授权。
3. 根据单位授权计算可绑定插件和可绑定资源。
4. 根据站点资源绑定计算问题清单。

第一阶段可以在前端组合计算，不新增数据库视图。等数据量上来后，再考虑新增单位工作台摘要视图。

## 交互路径

### 路径一：给新单位开通能力

1. 在单位管理页新增单位。
2. 点击“进入工作台”。
3. 在站点区域确认或绑定站点。
4. 在插件授权区域授权插件。
5. 在资源授权区域授权资源。
6. 在站点资源绑定区域选择站点，绑定 `plugin_code + usage_key -> resource`。
7. 观察问题数归零。

### 路径二：排查客户端拿不到资源

1. 进入单位工作台。
2. 查看顶部“待处理问题数”。
3. 在问题列表中定位原因。
4. 根据问题类型跳转到插件授权、资源授权或资源绑定区域。
5. 修复后重新加载并确认问题消失。

## 错误处理

页面应将技术错误和业务阻断分开：

- 技术错误：数据库请求失败、鉴权失败、网络失败，仍使用顶部错误提示。
- 业务阻断：未授权、未绑定、状态停用，在工作台内用问题提示展示。

停用授权和停用绑定要保留现有即时操作，但建议后续增加确认弹窗。本阶段如果不加弹窗，也必须在按钮文本和成功消息中说明影响对象。

## 测试策略

单元测试重点覆盖服务层计算：

- 根据单位过滤站点。
- 根据单位插件授权计算插件候选。
- 根据单位资源授权计算资源候选。
- 计算单位工作台摘要指标。
- 计算问题列表。

页面测试重点覆盖：

- 单位列表能进入工作台。
- 工作台能展示单位基础信息和摘要。
- 选择站点后能展示资源绑定。
- 未满足授权条件时显示明确原因。

回归验证：

- `npm test`
- `npm run build`
- 如涉及上线，继续使用 prod101 发布中心页面部署和健康检查流程。

## 分阶段实施建议

### 第一阶段：单位工作台骨架

- 新增单位工作台页。
- 单位管理页增加“进入工作台”动作。
- 工作台展示单位基础信息、摘要指标和四个区域。
- 复用现有新增授权、停用授权、保存绑定动作。

### 第二阶段：问题清单

- 增加单位级问题列表。
- 对站点、插件授权、资源授权、资源绑定做一致的业务状态提示。
- 增加快捷跳转和区域高亮。

### 第三阶段：视图与性能优化

- 当单位、站点、资源数量增长后，新增数据库视图或 RPC 返回单位工作台摘要。
- 引入分页、服务端筛选和增量加载。

## 推荐结论

下一步优先实施第一阶段“单位工作台骨架”。它不改变既有权限模型，不引入新的治理复杂度，但能显著降低资源中心开通和排障成本。
