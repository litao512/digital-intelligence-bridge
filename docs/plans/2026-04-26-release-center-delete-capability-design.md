# 发布中心资源删除能力设计

## 背景

DIB 发布中心已经支持客户端版本、插件版本、插件定义和发布资产的创建、上传、登记与 manifest 发布。但实际演练插件升级时，测试插件和临时资产只能新增，无法在页面内清理，导致发布中心很快积累测试记录。

当前软件仍处于开发阶段，没有正式上线历史负担。删除能力可以直接按“显式、可回滚认知、受约束”的方式设计，优先服务开发期发布演练和测试数据清理。

## 目标

为 DIB 发布中心补充通用资源删除能力：

1. 删除插件版本。
2. 删除插件定义。
3. 删除客户端版本。
4. 删除发布资产元数据。
5. 删除发布资产对应的 Storage 文件。

删除操作必须有确认提示，必须刷新页面数据，并且不能自动发布新的 manifest。

## 范围

纳入范围：

- 发布中心 Web 页面增加删除按钮和确认流程。
- 前端仓储层增加删除 API。
- 发布资产删除前检查引用关系。
- 发布资产支持同时删除 Storage 文件。
- 删除成功后刷新列表和 manifest 预览。
- 对关键删除路径增加测试。

不纳入范围：

- 数据库结构变更。
- 批量删除。
- 软删除或回收站。
- 自动发布 manifest。
- 从浏览器外部绕过页面读取认证令牌。
- 生产环境数据保留策略。

## 数据约束

发布中心当前数据库约束决定删除顺序必须明确。

- `plugin_versions.asset_id` 引用 `release_assets.id`，删除资产前必须先删除引用它的插件版本。
- `client_versions.asset_id` 引用 `release_assets.id`，删除资产前必须先删除引用它的客户端版本。
- `plugin_versions.package_id` 引用 `plugin_packages.id`，删除插件定义会级联删除该插件定义下的插件版本。
- 插件授权策略表引用 `plugin_packages.id`，删除插件定义会级联删除相关授权绑定。

因此发布资产删除采用阻断策略：只要仍被客户端版本或插件版本引用，就不允许删除资产元数据。

插件定义删除采用显式级联提示：确认文案必须说明会连带删除插件版本和相关授权绑定。

## 页面交互

### 插件发布页

插件版本表增加“操作”列，提供删除按钮。

删除插件版本时确认信息包含：

- 插件编码。
- 版本号。
- 发布渠道。
- 是否已发布。

删除后刷新插件版本列表，并刷新插件 manifest 预览。删除不会自动点击“发布 manifest”。

插件定义区域增加插件定义列表，展示插件编码、名称、入口类型、启用状态和操作按钮。

删除插件定义时确认信息必须说明：

- 会删除该插件定义。
- 会级联删除该插件定义下的插件版本。
- 会级联删除该插件定义关联的授权策略。
- 不会自动删除已上传的资产文件。

### 客户端发布页

客户端版本表增加“操作”列，提供删除按钮。

删除客户端版本时确认信息包含：

- 版本号。
- 发布渠道。
- 平台。
- 是否已发布。

删除后刷新客户端版本列表，并刷新客户端 manifest 预览。删除不会自动发布 manifest。

### 发布资产页

发布资产表增加“操作”列，提供删除按钮。

删除发布资产前先检查引用关系：

- 若存在插件版本引用，提示先删除对应插件版本。
- 若存在客户端版本引用，提示先删除对应客户端版本。

确认信息包含：

- 资产名称。
- 资产类型。
- bucket。
- storage path。
- SHA256。

发布资产删除分两种：

1. 只删除元数据：删除 `release_assets` 记录，保留 Storage 文件。
2. 同时删除文件：先删除 Storage 文件，再删除 `release_assets` 记录。

如果 Storage 文件删除失败，不继续删除资产元数据。如果 Storage 文件删除成功但资产元数据删除失败，页面显示修复提示，要求管理员重新上传或手工补齐元数据。

## 仓储层设计

新增或扩展以下仓储函数：

- `deletePluginVersion(id)`：删除单条插件版本。
- `deletePluginPackage(id)`：删除单个插件定义。
- `deleteClientVersion(id)`：删除单条客户端版本。
- `deleteReleaseAsset(id)`：删除单条发布资产元数据。
- `findReleaseAssetReferences(id)`：查询资产是否被客户端版本或插件版本引用。
- `deleteReleaseAssetObject(bucketName, storagePath)`：删除 Storage 文件。

仓储函数只封装 Supabase 调用和错误转换，不包含页面确认逻辑。

## 错误处理

删除失败时页面保留当前列表，并显示错误信息。

删除成功但刷新失败时，页面显示刷新错误，并允许管理员手动刷新。

删除版本后 manifest 预览必须重新拉取，避免继续展示已删除记录。

发布资产被引用时不调用删除接口，直接提示引用关系。

## 测试策略

仓储层测试优先覆盖：

- 插件版本删除调用正确的表和主键条件。
- 插件定义删除调用正确的表和主键条件。
- 客户端版本删除调用正确的表和主键条件。
- 发布资产删除调用正确的表和主键条件。
- 发布资产引用查询能汇总客户端版本和插件版本引用。
- Storage 文件删除调用正确的 bucket 和路径。

页面层测试覆盖关键交互：

- 删除确认取消时不调用仓储。
- 资产仍被引用时阻断删除。
- 删除成功后刷新列表。

## 风险与取舍

本次采用硬删除，因为项目尚未上线且用户明确需要清理测试资源。未来正式上线前，如需要审计和回滚，可以再引入软删除、审计日志或环境隔离。

本次不自动发布 manifest，是为了避免删除测试数据时误改客户端可见版本。管理员完成删除后，应手工检查 manifest 预览，再按需要点击发布。

本次使用浏览器确认弹窗即可满足开发期闭环。后续如果发布中心形成正式运营后台，可以改为统一的危险操作确认对话框。
