# 插件自动更新体验设计

## 背景

DIB 客户端已经具备发布中心检查、插件包下载、插件包预安装和启动时激活预安装插件的基础能力。当前问题是这些能力分散在首页和设置页，用户需要记住多个按钮和顺序，不利于日常使用。

本设计面向开发阶段的新软件，不考虑历史兼容负担，目标是让插件更新成为客户端的内建体验：发现更新后自动下载和预安装，用户只需要在合适时机重启客户端让插件生效。

## 目标

1. 用户可以方便地手动检查插件更新。
2. 客户端启动后可以静默检查插件更新。
3. 发现插件更新后自动下载和预安装。
4. 插件更新预安装成功后明确提示“重启后生效”。
5. 更新失败不阻塞客户端启动，不打断核心使用流程。

## 非目标

1. 不做后台定时轮询。
2. 不做插件热加载。
3. 不强制用户立即重启。
4. 不改变发布中心 manifest、插件包和启动激活的现有协议。

## 推荐策略

采用“自动下载和预安装，提示重启后生效”的策略。

流程为：

1. 检查发布中心插件 manifest。
2. 如果存在当前站点可用插件包，自动下载到缓存目录。
3. 下载成功后自动生成预安装目录。
4. 如果有插件包完成预安装，在首页状态区和设置页状态区显示“需要重启”。
5. 下次启动时沿用现有激活流程，把预安装插件切换到运行时插件目录。

## 触发入口

### 手动触发

首页“检查更新”升级为插件更新快捷入口。用户点击后，客户端执行完整流程：检查、下载、预安装、刷新本地插件状态。

设置页仍保留底层操作按钮，便于开发和排障，但普通用户主要从首页完成插件更新。

### 启动后静默触发

主窗口完成初始化后延迟几秒执行一次静默检查。静默检查只更新状态，不弹出阻塞式对话框。

如果没有可用插件包，不打扰用户。如果有插件包完成预安装，提示重启后生效。

### 事件触发

站点信息保存成功、初始化本机插件完成后，可以复用同一套插件更新编排流程。这样站点授权变化后，用户无需记住手工执行多个步骤。

## 架构

新增客户端侧插件更新编排服务（Plugin Update Orchestrator），封装现有 `IReleaseCenterService` 的检查、下载和预安装调用。首页、设置页和启动流程只调用这个编排服务，不再各自拼接流程。

编排服务保持轻量，不持久化复杂状态。它返回一次更新流程的结果，包括检查摘要、下载摘要、预安装摘要、是否需要重启、失败详情和时间戳。

## 数据流

1. UI 或启动流程调用编排服务。
2. 编排服务确认发布中心已配置。
3. 调用 `CheckForUpdatesAsync` 获取客户端、插件和授权摘要。
4. 调用 `DownloadAvailablePluginPackagesAsync` 下载可用插件包。
5. 如果下载成功，调用 `PrepareCachedPluginPackagesAsync` 生成预安装目录。
6. 返回统一结果给调用方。
7. 调用方刷新本地插件清单，并根据结果更新提示。

## 错误处理

1. 发布中心未配置：返回“未配置”，不抛出到 UI。
2. 站点信息不足：提示先完善站点信息。
3. 检查失败：保留失败详情，停止后续下载和预安装。
4. 下载失败：保留失败详情，停止预安装。
5. 预安装失败：提示排查，不标记为需要重启。
6. 静默检查失败：记录日志并更新轻量状态，不弹出阻塞提示。

## 用户界面

首页状态区显示最近插件更新流程结果：

- 没有更新：显示最近检查时间。
- 已预安装：显示“需要重启”，说明插件更新将在重启后生效。
- 失败：显示“需要排查”，展示简短错误详情。

设置页继续显示检查、下载、预安装的详细摘要，方便开发阶段排障。

## 测试策略

1. 编排服务单元测试覆盖成功路径：检查成功、下载成功、预安装成功并返回需要重启。
2. 编排服务单元测试覆盖无服务、未配置、检查失败、下载失败、预安装失败。
3. 首页 ViewModel 测试覆盖“检查更新”会调用编排服务并刷新重启提示。
4. 启动触发测试覆盖主窗口初始化后会调用静默检查入口，且失败不阻塞启动。

## 验收标准

1. 点击首页“检查更新”后，插件包可自动下载和预安装。
2. 插件预安装成功后，首页明确提示重启后生效。
3. 客户端启动后会静默检查插件更新。
4. 没有更新或检查失败不会阻塞主界面打开。
5. 现有设置页手动检查、下载、预安装能力继续可用。
