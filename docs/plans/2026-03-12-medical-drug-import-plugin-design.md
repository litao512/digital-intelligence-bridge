# 医保药品导入同步工具设计

## 1. 背景

当前项目已经具备桌面应用骨架、配置系统、托盘能力，以及基于 `plugins/` 的插件发现占位逻辑，但尚未具备真实的外部插件加载、生命周期管理和服务注入能力。

本次要实现的功能不是简单页面扩展，而是一个复合型工具模块：

- 读取固定模板 Excel
- 将原始数据写入 PostgreSQL 中间表
- 执行字段清洗与业务表合并
- 将本批次增量同步到 SQL Server
- 在桌面端展示执行过程、结果摘要和错误信息

结合当前代码基线与已有数据库结构，第一版不适合直接做外部 DLL 热插拔插件，更合适的方式是先实现为内置 Avalonia 工具模块，同时为后续插件化演进预留服务边界。

## 2. 目标

### 2.1 功能目标

实现一个“医保药品导入同步工具”模块，支持人工触发以下流程：

1. 选择固定模板 Excel 文件
2. 对工作表和表头执行预检
3. 流式导入 PostgreSQL
4. 清洗并合并到 `biz.drug_catalog`
5. 将本批次增量同步到 SQL Server `dbo.yb_药品目录`
6. 输出批次摘要、错误样例和同步结果

### 2.2 非目标

第一版明确不包含以下能力：

- 外部 DLL 插件动态加载
- 定时后台同步
- 可配置列映射模板
- 全量结果明细预览
- 自动回滚跨库事务
- 签名校验、插件隔离、安全沙箱

## 3. 已确认约束

### 3.1 交互约束

- 同步方式为人工触发
- Excel 模板为固定模板
- 运行机器内存约 4 GB，不能采用全量加载 Excel 的方式

### 3.2 项目约束

- 当前项目为 Avalonia + Prism + DryIoc 的桌面应用
- 当前插件体系仅有目录扫描与状态展示，没有真实插件契约
- 当前更适合新增内置模块，而不是直接构建外部插件框架

### 3.3 数据约束

已确认 PostgreSQL 存在相关表：

- `etl.drug_import_raw`
- `etl.drug_import_clean`
- `etl.drug_import_error`
- `etl.drug_catalog_sync_outbox`
- `biz.drug_catalog`
- `biz.drug_change_log`
- `biz.drug_code_relation`

已确认 SQL Server 存在相关表：

- `dbo.yb_药品目录`
- `dbo.yb_同步记录`

因此第一版应直接对接现有数据库结构，而不是重新设计一套新表。

## 5. 总体方案

推荐方案为“分层模块化实现”：

- `View/ViewModel` 负责用户交互、批次展示、结果摘要
- `ExcelImportService` 负责 Excel 流式读取与模板解析
- `DrugImportPipelineService` 负责 PostgreSQL 侧入库、清洗、合并、变更记录
- `SqlServerSyncService` 负责将本批次增量同步到 SQL Server

### 5.1 为什么不直接做外部 DLL 插件

- 当前应用尚无真实插件契约
- 本功能已包含 UI、Excel ETL、跨库同步，复杂度较高
- 若同时引入动态加载、生命周期、异常隔离，会明显增加首版风险

因此，第一版应先实现为内置工具模块，待链路跑通后，再把核心服务抽象成插件 API。

## 6. 架构分层

### 6.1 UI 层

新增一个工具页面，提供以下能力：

- 选择 Excel 文件
- 执行模板预检
- 启动导入
- 启动 SQL Server 同步
- 查看批次统计
- 查看错误样例和日志摘要

UI 层不承载实际业务规则，不保存大批量行数据。

### 6.2 应用服务层

#### `ExcelImportService`

职责：

- 校验工作表是否存在
- 校验表头是否匹配固定模板
- 以流式方式枚举每一行数据
- 生成标准化的中间行对象

关键要求：

- 不整表加载
- 不在内存中保留完整明细
- 能按 sheet 输出有效行计数与预检信息

#### `DrugImportPipelineService`

职责：

- 创建 `batch_id`
- 将原始行写入 `etl.drug_import_raw`
- 标准化字段后写入 `etl.drug_import_clean`
- 将异常写入 `etl.drug_import_error`
- 合并到 `biz.drug_catalog`
- 写入 `biz.drug_change_log`
- 处理 `关联关系表` 到 `biz.drug_code_relation`

#### `SqlServerSyncService`

职责：

- 基于 `batch_id` 或 `source_batch_id` 找出本批次受影响的 `biz.drug_catalog`
- 按业务键 upsert 到 `dbo.yb_药品目录`
- 将同步摘要写入 `dbo.yb_同步记录`
- 对失败提供可重试能力

### 6.3 数据访问层

建议分别封装：

- PostgreSQL 仓储
- SQL Server 仓储

避免在 ViewModel 中直接拼装 SQL。

## 7. 数据流设计

### 7.1 预检阶段

用户选择文件后，先执行轻量预检：

1. 检查文件存在且可读
2. 检查 4 个工作表是否存在
3. 检查表头是否匹配
4. 统计各工作表有效行数
5. 输出预检报告

预检阶段不写数据库。

### 7.2 导入阶段

正式导入时：

1. 生成 `batch_id`
2. 按工作表逐行流式读取
3. 将每一行原始数据写入 `etl.drug_import_raw`
4. 将标准化后的结果写入 `etl.drug_import_clean`
5. 对异常行写入 `etl.drug_import_error`

该阶段完成后得到完整的批次落库痕迹。

### 7.3 合并阶段

使用 `etl.drug_import_clean` 中的有效记录：

1. 按 `药品代码` 作为主业务键归并
2. upsert 到 `biz.drug_catalog`
3. 对本批次变更写入 `biz.drug_change_log`
4. 对 `关联关系表` 写入 `biz.drug_code_relation`

### 7.4 SQL Server 同步阶段

以 `batch_id` 或 `source_batch_id` 为范围：

1. 查询本批次影响的 `biz.drug_catalog`
2. 将其 upsert 到 `dbo.yb_药品目录`
3. 将同步摘要写入 `dbo.yb_同步记录`

第一版不依赖 `etl.drug_catalog_sync_outbox`，因为该表当前为空，现有链路未形成闭环。

## 8. 字段与业务键设计

### 8.1 主同步键

主键使用 Excel 的 `药品代码`：

- PostgreSQL：`biz.drug_catalog.drug_code`
- SQL Server：`dbo.yb_药品目录.药品编码`

### 8.2 编码关系

`关联关系表` 用于构建：

- 当前码
- `goods_id`
- 曾用码
- 曾用 `goods_id`

该关系建议写入 `biz.drug_code_relation`，后续用于历史编码追踪和辅助匹配。

### 8.3 主要字段对齐

已确认以下字段存在明显映射关系：

- `drug_name_cn -> 中文名称`
- `drug_name_en -> 英文名称`
- `pinyin_code -> 拼音助记码`
- `wubi_code -> 五笔助记码`
- `dosage_form -> 剂型`
- `specification -> 规格`
- `packaging_material -> 包装材质`
- `min_pack_qty -> 最小包装数量`
- `min_dose_unit -> 最小制剂单位`
- `min_pack_unit -> 最小包装单位`
- `drug_base_code -> 药品本位码`
- `marketing_status -> 市场状态`
- `approval_no -> 国药准字`

### 8.4 未提供字段策略

SQL Server 中若存在 Excel 本次未提供的字段，如：

- `收费类别`
- `收费项目等级`
- `处方药标志`
- `每次用量`
- `使用频次`
- `用法`

第一版建议策略：

- Excel 未提供值时，不覆盖既有非空值
- 仅更新本次明确提供或能稳定推导的字段

## 9. 失败恢复与幂等

### 9.1 批次原则

所有导入记录必须绑定 `batch_id`，作为追踪和重试的最小单位。

### 9.2 失败处理

#### Excel 解析失败

- 保留已写入的 `raw/clean/error`
- 批次标记为导入失败
- 允许重新选择文件再导入，生成新批次

#### PostgreSQL 合并失败

- 不进入 SQL Server 同步阶段
- 保留中间表痕迹，便于排错

#### SQL Server 同步失败

- 不回滚 PostgreSQL 结果
- 批次标记为“待重试同步”
- 后续允许只重跑该批次同步步骤

### 9.3 幂等要求

- SQL Server 必须按业务键执行 upsert
- 同一批次重复触发同步时，不得产生重复数据

## 13. 结论

当前最合理的实施路径不是“直接开发一个外部 DLL 插件”，而是：

1. 先以内置 Avalonia 工具模块实现医保药品导入同步能力
2. 直接复用 PostgreSQL 与 SQL Server 现有表结构
3. 用流式解析满足低内存环境
4. 以 `batch_id` 实现完整追踪、失败保留与同步重试
5. 待链路稳定后，再抽象成更通用的插件能力

## 15. 架构调整说明（2026-03-18）

截至 2026 年 3 月 18 日，本项目已经新增第一阶段外部插件宿主能力，因此“医保药品导入”后续不再继续沿着内置模块扩张，而是转为以下双轨结构：

- 现有内置导入工具仍可作为业务实现参考
- 新的 `MedicalDrugImport.Plugin` 作为第一个外部插件样例，优先验证 DLL 加载、菜单接入与页面承载

当前外部插件阶段已完成：

- `Plugin.Abstractions` 契约项目
- `Plugin.Host` 宿主项目
- `plugins-src/MedicalDrugImport.Plugin` 样例插件项目
- `plugins/MedicalDrugImport/` 运行时插件目录
- 主程序启动时的插件发现、加载、菜单注入与页面承载
- 最小错误边界与 `minHostVersion` 兼容性校验

因此，医保药品业务链路的下一阶段重点应调整为：

1. 把 Excel / PostgreSQL / SQL Server 业务服务逐步从内置工具抽离到外部插件内部
2. 宿主继续只负责清单、加载、菜单、页面承载和失败隔离
3. 内置模块后续只保留过渡用途，避免与外部插件双线长期并存
