# 插件中心与菜单分组设计

## 背景

DIB 客户端已经支持运行时加载插件菜单。当前问题是插件菜单和首页、资源中心等内置菜单混在同一个导航列表中，用户难以区分“系统内置能力”和“业务插件能力”。随着插件数量增加，主导航会变长，插件生命周期管理也缺少独立入口。

本设计目标是建立清晰的信息架构：首页看概况，资源中心管资源权限，插件中心管插件生命周期，左侧插件区进入业务插件页面。

## 推荐方案

采用“轻量分组 + 插件中心”的第一版方案：

```text
工作台
  首页
  资源中心
  插件中心

插件
  就诊登记
  药品导入
  其他插件入口

系统
  设置
```

第一版只做一级分组，不做插件二级折叠。这样能快速解决菜单混杂问题，同时不把插件协议和导航模型过早复杂化。

## 目标

1. 左侧导航明确区分内置功能、插件功能和系统功能。
2. 新增“插件中心”作为内置页面入口。
3. 插件加载出来的菜单统一归入“插件”分组。
4. 首页只保留插件摘要和状态，不作为插件管理主入口。
5. 设置可以继续固定在底部，但模型上归入系统功能。

## 非目标

1. 不做插件二级折叠菜单。
2. 不做插件搜索。
3. 不做启用、禁用、卸载插件。
4. 不改变插件菜单协议。
5. 不把插件业务入口全部藏到插件中心。

## 导航结构

### 工作台

工作台包含客户端内置的日常入口：

- 首页
- 资源中心
- 插件中心

资源中心继续负责资源发现、授权资源、资源申请和审批状态。

插件中心负责插件安装状态、版本状态、更新状态、待重启状态和失败详情。

### 插件

插件分组承载所有运行时插件暴露的业务菜单。第一版按插件菜单 `Order` 排序展示，不做二级层级。

如果未来单个插件提供多个入口，第二版再升级为按插件名称折叠：

```text
插件
  就诊登记
    患者登记
    登记记录
```

### 系统

系统分组包含设置等低频维护入口。现阶段设置按钮可以继续固定在左下角，保持桌面软件常见布局。

## 插件中心第一版功能

插件中心第一版做轻量管理视图：

1. 插件总览：已安装数量、授权插件数量、待重启数量、失败数量。
2. 插件列表：插件名称、插件 ID、当前版本、状态。
3. 操作区：检查更新、重启 DIB。
4. 状态详情：最近检查结果、最近预安装结果、失败详情。

插件中心复用现有 `HomeDashboardViewModel` 已具备的插件状态读取能力和 `IPluginUpdateOrchestrator` 自动更新能力。第一版不新增复杂持久化模型。

## 菜单模型

`MenuItem` 增加分组字段：

- `Workspace`
- `Plugin`
- `System`

视图模型继续维护一个 `MenuItems` 集合，但可以新增分组视图模型，例如 `MenuSections`。UI 绑定分组后的集合，按分组标题渲染。

插件菜单加载时统一设置为 `Plugin` 分组。首页、资源中心、插件中心设置为 `Workspace` 分组。设置设置为 `System` 分组。

## 用户体验

1. 用户打开客户端后，左侧先看到稳定的内置工作台入口。
2. 插件业务入口出现在独立的“插件”分组下，不再和内置功能混排。
3. 插件中心提供插件生命周期总览，帮助用户理解哪些插件已生效、哪些待重启、哪些需要排查。
4. 首页保留简洁摘要，避免承载过多管理功能。

## 错误处理

1. 插件中心读取本地插件状态失败时，显示失败摘要，不影响主窗口。
2. 自动检查更新失败时，显示最近失败详情，不阻塞插件列表展示。
3. 没有插件时，插件分组显示空状态或隐藏插件菜单项，插件中心仍可进入。

## 测试策略

1. 菜单初始化测试：内置菜单分别进入正确分组。
2. 插件菜单测试：外部插件菜单统一进入“插件”分组。
3. 插件中心导航测试：点击插件中心打开对应标签页。
4. 插件中心 ViewModel 测试：能读取运行时插件、预安装插件和授权插件摘要。
5. 构建测试：Avalonia XAML 编译通过。

## 验收标准

1. 左侧导航显示“工作台”和“插件”分组。
2. 首页、资源中心、插件中心位于“工作台”。
3. 插件暴露的业务菜单位于“插件”。
4. 插件中心可以打开，并显示插件状态列表和更新操作入口。
5. 现有插件页面仍可从左侧插件分组直接打开。
