# DIB 中心资源中心一期设计

## 背景

DIB 当前已经具备发布中心、站点管理、插件发布、客户端发布和资源中心最小运行时能力。资源中心现有实现已经覆盖客户端入口、资源发现、资源申请、授权资源缓存、站点级资源绑定和相关 RPC，但管理端还没有形成完整闭环。

随着 DIB 从医疗场景继续扩展到疾控、医保、民政、大数据局等行业，发布中心不再只是发布插件和客户端的工具，而是在演进为 DIB 的统一控制平面。资源中心也需要从“站点可以申请资源”升级为“以单位为治理主体，统一管理站点、插件、资源和授权绑定”。

本设计面向资源中心一期演进，重点解决三个问题：

1. 站点、单位、插件、资源之间的主关系是什么。
2. 同一个插件在不同站点连接不同资源时，授权和绑定如何表达。
3. 标签是否引入，以及在当前阶段承担什么职责。

## 核心结论

资源中心一期采用以下原则：

```text
单位授权决定可选范围
站点绑定决定实际使用
```

单位是权限和治理主体，站点是部署实例。站点必须归属某个单位，插件和资源先在单位层面获得授权，站点只能在所属单位允许范围内建立实际运行绑定。

运行时仍然保留站点级资源绑定：

```text
站点 + 插件 + usage_key -> 资源
```

这条绑定不直接代替单位授权，而是在单位授权范围内确定最终连接哪个资源。

## 目标

一期目标如下：

1. 建立正式单位模型，支撑医院、疾控中心、医保局、民政局、大数据局等主体。
2. 让站点绑定到单位，由单位信息限定站点可用插件和可访问资源。
3. 让资源拥有明确所属单位，用于表达资源归属、维护责任、审批责任和数据责任。
4. 增加单位插件授权，表达某单位可以使用哪些插件。
5. 增加单位资源授权，表达某单位可以访问哪些资源。
6. 保留并强化站点级资源绑定，用于决定插件运行时实际连接的资源。
7. 引入轻量业务标签，用于筛选、推荐、展示和辅助理解，不作为硬权限依据。
8. 在 DIB 中心补齐资源管理、单位授权、站点资源绑定和申请审批的管理闭环。

## 非目标

一期暂不做以下能力：

1. 不做复杂策略引擎。
2. 不做多级组织树和跨层级继承。
3. 不做标签驱动的自动授权。
4. 不做跨单位资源市场化分发。
5. 不做资源自动发现和自动接入。
6. 不要求一次性重命名 `dib-release-center` 工程。

## 领域模型

### 单位

单位是资源中心的治理主体，代表真实组织或机构。

示例：

```text
A 医院
B 疾控中心
C 医保局
D 民政局
E 大数据局
```

建议新增正式单位表：

```text
organizations
```

建议核心字段：

```text
id
code
name
organization_type
business_tags
status
created_at
updated_at
```

其中 `organization_type` 可表达医院、疾控、医保、民政、大数据局等主体类型。`business_tags` 是轻量业务标签，用于辅助分类和筛选。

### 站点

站点是 DIB 客户端或运行环境的部署实例。站点不应单独成为资源权限主体，而应归属于单位。

建议在现有站点表上增加：

```text
organization_id
business_tags
```

含义：

```text
organization_id：该站点属于哪个单位
business_tags：该站点当前启用或关注的业务场景
```

示例：

```text
A 医院门诊站点 -> A 医院
A 医院测试站点 -> A 医院
B 疾控数据上报站点 -> B 疾控中心
```

### 插件

插件是能力模块。插件声明自己需要哪些资源用途，但不直接决定具体连接哪个资源。

插件需要表达两个层次：

```text
插件本身：patient-registration
插件资源用途：registration-db、insurance-api、archive-db
```

`usage_key` 是插件内部的资源用途名。它不是资源 ID，也不是数据库名称，而是插件声明的资源槽位。

### 资源

资源是真实外部能力，例如数据库、接口、文件服务、模型服务或消息队列。

建议资源具备明确所属单位：

```text
resources.owner_organization_id
```

它用于表达：

```text
资源由谁提供
资源由谁维护
资源审批责任属于谁
资源数据责任属于谁
```

当前已有 `resources.owner_organization_name`，它是文本形式的弱模型。一期可以保留兼容字段，同时新增正式外键字段，逐步迁移。

## 授权模型

资源中心一期分为单位授权和站点绑定两层。

### 单位插件授权

单位插件授权回答：

```text
这个单位有没有资格使用某个插件？
```

建议新增：

```text
organization_plugin_permissions
```

核心字段：

```text
organization_id
plugin_code
status
granted_by
granted_at
expires_at
```

### 单位资源授权

单位资源授权回答：

```text
这个单位有没有资格访问某个资源？
```

建议新增：

```text
organization_resource_permissions
```

核心字段：

```text
organization_id
resource_id
status
granted_by
granted_at
expires_at
```

### 站点资源绑定

站点资源绑定回答：

```text
这个站点上的这个插件，在某个资源用途上，实际连接哪个资源？
```

现有 `resource_bindings` 应继续保留，并被解释为站点运行绑定。

核心关系：

```text
site_id
plugin_code
usage_key
resource_id
status
```

约束原则：

```text
绑定前必须确认站点所属单位已获得插件授权
绑定前必须确认站点所属单位已获得资源授权
同一站点、同一插件、同一 usage_key 只能绑定一个激活资源
```

示例：

```text
A 医院门诊站点 + patient-registration + registration-db -> A-HIS-正式库
A 医院测试站点 + patient-registration + registration-db -> A-HIS-测试库
B 医院门诊站点 + patient-registration + registration-db -> B-HIS-正式库
```

这解决了同一插件在不同站点连接不同资源的问题。

## 资源发现与运行时流程

### 候选资源发现

客户端或管理端需要发现某站点某插件某用途可以绑定哪些资源时，流程为：

```text
站点 -> 所属单位
单位 -> 已授权插件
单位 -> 已授权资源
插件 + usage_key -> 所需资源类型或能力
资源类型、能力、标签 -> 辅助筛选候选资源
```

结果是候选资源列表，而不是最终运行资源。

### 实际资源获取

插件运行时需要获取实际资源时，流程为：

```text
站点 + 插件 + usage_key -> resource_bindings -> 资源连接信息
```

服务端在返回资源前应校验：

```text
站点存在且绑定单位
单位拥有该插件授权
单位拥有该资源授权
站点存在激活的资源绑定
资源和密钥状态有效
```

### 申请审批流程

站点发起资源申请时，申请不应只表达“站点想要资源”，还应携带单位上下文。

建议申请审批逻辑为：

```text
站点提交申请
DIB 中心识别站点所属单位
管理员审批单位是否可访问资源
必要时创建单位资源授权
再为站点建立具体资源绑定
客户端刷新授权资源
```

如果单位已经具备资源授权，审批可以只处理站点绑定。如果单位尚未具备资源授权，审批应先处理单位资源授权，再处理站点绑定。

## 标签策略

一期可以引入标签，但标签只作为轻量元数据。

可标注对象：

```text
单位：医疗、疾控、医保、民政、大数据局
站点：门诊、随访、报表、数据上报
插件：患者登记、医保结算、公共卫生
资源：HIS、医保接口、慢病管理、人口数据
```

标签用途：

```text
筛选资源
推荐候选资源
辅助管理员理解适用场景
减少选择成本
支持后续统计分析
```

标签不用于：

```text
自动授权
绕过单位插件授权
绕过单位资源授权
绕过站点资源绑定
```

一期可先采用 JSON 字段保存业务标签，避免过早引入复杂标签字典和多张关联表。后续如果标签成为重要治理能力，再升级为正式标签字典。

## DIB 中心管理端能力

一期建议在 DIB 中心补齐以下管理页面或管理区块。

### 单位管理

能力：

```text
新增单位
编辑单位名称、类型、状态和业务标签
查看单位下属站点
查看单位已授权插件
查看单位已授权资源
```

### 资源管理

能力：

```text
新增资源
编辑资源所属单位
编辑资源类型、能力、状态和业务标签
维护资源连接配置和密钥
查看资源被哪些单位授权
查看资源被哪些站点绑定
```

### 单位授权管理

能力：

```text
给单位授权插件
撤销单位插件授权
给单位授权资源
撤销单位资源授权
查看授权状态和有效期
```

### 站点绑定管理

能力：

```text
查看站点所属单位
查看站点可用插件
查看站点可绑定资源
为站点插件 usage_key 绑定资源
停用或替换站点资源绑定
```

### 申请审批

能力：

```text
查看站点资源申请
识别申请站点所属单位
判断单位是否已有资源授权
批准单位资源授权
批准站点资源绑定
拒绝申请并记录原因
```

## 客户端影响

客户端资源中心当前已有“可申请资源、已授权资源、待审批申请”的基本界面，一期应尽量保持客户端体验稳定。

客户端主要变化：

```text
资源发现结果由站点所属单位限定
资源申请自动带上单位上下文
已授权资源仍然按站点实际绑定返回
授权资源缓存继续按插件和 usage_key 使用
```

客户端不需要直接理解完整单位授权模型。客户端只需要知道当前站点可申请什么、已授权什么、申请状态如何。

## 数据兼容与迁移

当前资源中心运行时已经有以下关键表：

```text
resources
resource_bindings
resource_applications
resource_plugins
resource_secrets
```

一期建议新增：

```text
organizations
organization_plugin_permissions
organization_resource_permissions
```

并增强：

```text
sites.organization_id
sites.business_tags
resources.owner_organization_id
resources.business_tags
```

兼容策略：

```text
保留 resources.owner_organization_name 作为历史兼容字段
保留现有 resource_bindings 结构，逐步增加单位授权校验
现有 RPC 可先在内部补单位过滤，不立即改变客户端契约
历史站点可先绑定到默认单位或待分配单位
历史资源可先绑定到默认资源所属单位或待确认单位
```

## RPC 调整方向

现有 RPC 可以沿用名称，但内部语义需要升级。

### 获取站点已授权资源 RPC

`get_site_authorized_resources` 升级后语义：

```text
根据 site_id 找到站点和所属单位
读取站点级 resource_bindings
校验单位插件授权
校验单位资源授权
返回当前站点实际可用资源
```

### 发现站点资源 RPC

`discover_site_resources` 升级后语义：

```text
根据 site_id 找到站点和所属单位
根据单位插件授权和单位资源授权筛选候选资源
结合插件、usage_key、资源类型、能力、标签进行推荐
返回可申请资源、已授权资源和待审批申请
```

### 申请站点资源 RPC

`apply_site_resource` 升级后语义：

```text
记录站点资源申请
自动关联站点所属单位
审批时根据单位授权状态决定是否补单位资源授权
审批通过后建立或更新站点级 resource_bindings
```

## 验证要点

设计落地后，至少验证以下场景：

1. A 单位和 B 单位都使用同一个插件，但分别绑定不同数据库。
2. 一个单位有多个站点，同一插件的同一 `usage_key` 在不同站点绑定不同资源。
3. 站点尝试绑定未被所属单位授权的资源时被拒绝。
4. 站点尝试使用所属单位未授权的插件时无法获取资源。
5. 客户端已授权资源接口仍能返回插件运行所需连接信息。
6. 客户端资源中心仍能展示可申请资源、已授权资源和待审批申请。
7. 标签缺失或标签不匹配时，不影响已有硬授权判断。

## 后续演进

一期稳定后，可以继续演进：

1. 建立正式标签字典和标签关联表。
2. 支持组织层级和上级单位授权继承。
3. 支持资源共享审批和跨单位授权协议。
4. 支持插件声明资源需求清单，让 `usage_key`、资源类型、能力形成更严格契约。
5. 支持策略引擎，用于更复杂的行业权限规则。
6. 将 `dib-release-center` 的产品名称和导航逐步升级为 DIB 中心。

## 当前决策摘要

资源中心一期不把复杂性推给客户端，也不让标签成为权限核心。系统先建立清晰的治理骨架：

```text
单位是权限主体
站点是部署实例
插件声明资源用途
资源归属单位
单位授权限定范围
站点绑定决定实际运行资源
标签只做辅助分类
```

这套模型既能兼容当前资源中心最小运行时，又能支撑后续面向医疗、疾控、医保、民政、大数据局等多行业扩展。
