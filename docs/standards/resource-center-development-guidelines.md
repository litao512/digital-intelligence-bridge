# 资源中心开发约定

## 适用范围
- 本文适用于资源中心相关的宿主服务、ViewModel、插件声明协议、后台 RPC 契约和对应测试。

## 契约映射约定
- 资源中心相关的 RPC/JSON 返回对象必须显式声明字段映射，不依赖 C# 属性名或 record 参数名的默认推断。
- 当后台字段使用 `camelCase`、`snake_case` 或其他与本地命名不一致的写法时，统一使用 `JsonPropertyName` 固化映射关系。
- 新增或修改 RPC 返回结构时，先补契约测试，再改生产代码，避免出现“反序列化成功但值为默认值”的静默错误。

## 实现顺序约定
- 资源中心功能默认按以下顺序推进：
  1. 服务层契约测试
  2. ViewModel 行为测试
  3. 视图绑定
  4. 手动回归
- 不要从 UI 表现层直接倒推服务契约。资源中心的主要风险在接口契约和状态同步，优先锁定服务层行为。

## 接口扩展约定
- `IReleaseCenterService` 扩展接口时，第一步先全局搜索所有 fake、stub、test double，并补齐默认实现。
- 测试桩未补齐前，不要开始判断业务逻辑是否正确。优先消除接口签名带来的编译噪音，再看真实失败用例。
- 资源中心相关测试桩默认返回可识别的占位结果，例如固定 `ApplicationId`、固定状态值，避免后续断言含糊。

## 验证约定
- 资源中心功能验证继续遵循仓库已有的 Avalonia 串行约束：
  - `dotnet build` 和 `dotnet test` 不并行执行
  - 本地验证优先使用 `-m:1`
  - 推荐先 `build`，再 `test`
- 当资源中心改动涉及 UI、服务和测试工程时，至少覆盖：
  - 目标服务测试
  - 目标 ViewModel 测试
  - 主项目串行构建
  - `git diff --check`

## 常见风险
- 后台字段名与本地字段名不一致，导致反序列化结果静默失真。
- `IReleaseCenterService` 变更后，散落在多个测试文件里的测试桩未同步更新，先卡编译。
- UI 已接通但服务契约未锁定，导致问题表面出现在界面，根因却在 RPC 输入输出。
