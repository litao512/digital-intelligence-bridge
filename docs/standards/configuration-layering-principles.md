# 配置分层原则

## 适用范围
- 本文适用于 DIB 宿主、资源中心、运行时插件以及与配置来源相关的实现、排障和后续设计。

## 核心结论
- 本系统是全新设计系统，不保留历史兼容负担。
- 正式生产环境下，插件业务资源配置不再通过环境变量或 `plugin.settings.json` 持有。
- 正式生产资源唯一来源应为：`资源中心 -> 宿主缓存 -> 插件读取`。
- 正式设计里，插件不存在“生产配置加载顺序”，只存在“正式资源来源”。

## 一、主程序负责什么
主程序只负责两类配置：

1. 宿主部署控制
- 例如：
  - `DIB_CONFIG_ROOT`
  - 发布中心/资源中心地址
  - 日志目录
  - 运行渠道
  - 插件扫描根目录
  - 宿主能力开关

2. 本机站点状态
- 例如：
  - `SiteId`
  - `SiteName`
  - `SiteRemark`
  - 托盘行为偏好
  - UI 本机偏好

单位归属不属于客户端本机状态。客户端只持有 `SiteId`，站点所属单位由 DIB 中心的 `sites.organization_id` 管理。

结论：
- 主程序配置负责“宿主怎么运行”和“这台机器是谁”。
- 主程序配置不负责插件生产资源连接。

## 二、资源中心负责什么
资源中心是正式生产资源的唯一真相源。

资源中心必须负责：
- 资源定义
- 连接参数
- 敏感凭据
- 单位/站点/插件绑定关系
- 授权状态
- 审批状态

典型资源包括：
- PostgreSQL
- SqlServer
- Supabase
- HTTP 服务
- OCR 服务
- 人脸识别服务

结论：
- 凡是插件需要访问的正式外部资源，都应进入资源中心。

## 三、插件负责什么
插件只负责两件事：

1. 声明资源需求
- 通过 `plugin.json` 中的 `resourceRequirements`

2. 消费宿主下发资源
- 通过 `IPluginHostContext`
- 按 `usageKey` 读取运行时资源

插件不再负责：
- 定义正式生产连接来源
- 自己拼接生产数据库连接
- 通过环境变量管理生产资源

## 四、插件本地配置的定位
`plugin.settings.json` 不是正式生产配置来源，也不是生产回退路径。

它只允许用于：
- 非敏感界面偏好
- 本地打印样式
- 本地非中心化功能开关
- 开发/联调专用默认值
- 本地调试辅助参数

不允许用于：
- 生产数据库连接
- 生产 API Key
- 生产 Token
- 生产 Supabase 凭据
- 生产 OCR/人脸服务地址与密钥

结论：
- `plugin.settings.json` 的定位是“开发配置和非敏感本地配置”，不是正式生产配置，也不是正式生产回退。

## 五、环境变量的定位
环境变量只保留在宿主部署层，不再作为插件正式业务资源入口。

允许保留的环境变量用途：
- 指定配置根目录
- 指定宿主级部署参数
- 测试隔离
- 运维级临时覆盖

不再推荐的用途：
- 插件数据库连接
- 插件外部服务地址
- 插件生产密钥

结论：
- 插件业务资源环境变量方案属于废弃设计，不再作为正式方向继续扩展。

## 六、正式优先级

### 6.1 主程序
主程序配置优先级为：

1. 安装目录 `appsettings.json`
2. 用户目录 `appsettings.json`
3. 宿主环境变量

后加载覆盖先加载。

### 6.2 插件
正式生产环境下，插件资源来源只有一个：

1. 宿主授权资源

开发/联调时，才允许显式启用本地开发配置。
插件环境变量和本地文件都不再作为正式生产层级。

## 七、实施约束
- 新插件默认不再设计“环境变量读取生产连接”的入口。
- 新插件默认不再设计“本地配置文件持有生产连接串”的入口。
- 插件如需开发模式，应通过显式开发开关进入，不允许与正式生产路径混用。
- 旧插件迁移时，应优先把生产连接迁入资源中心，再将本地文件降级为开发配置。
- 文档、代码注释和排障说明中，不再把插件环境变量作为推荐生产做法。
- 文档、代码注释和排障说明中，不再出现“插件生产配置加载顺序”这一表述，应统一改为“插件正式资源来源”。

## 八、排障顺序
当出现“插件没有按预期读取配置”时，排障顺序应为：

1. 先确认宿主是否拿到了授权资源
2. 再确认宿主缓存是否可读
3. 再确认插件是否按 `usageKey` 正确读取
4. 如果处于开发模式，再看本地开发配置

不要优先怀疑插件环境变量，因为它不再是正式设计路径。
